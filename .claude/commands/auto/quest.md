# /auto:quest — 闯关大纲设计

> **输入一份需求，输出一套精确到代码动作级别的渐进式闯关教案**

`/auto:quest` 将复杂需求拆解为原子化的 Quest Map。核心差异化：**先深度分析代码，再设计任务**，而非凭空规划。

---

## 核心价值

- **精确到代码级**：每个 Quest 列出具体的文件操作（新增/修改/删除）
- **依赖感知排序**：基于 import 分析确定 Quest 顺序，不会出现依赖倒置
- **可执行验收**：每条验收标准都是可粘贴到终端的命令
- **自验证防护**：输出前逐条评分，低于 7/10 的 Quest 必须修改

---

## 执行流程

```
用户输入需求
      |
[第1步] 需求理解 + 信息完整性检查
      |  → 信息充足则不反问，直接进入分析
      |
[第2步] 深度代码分析（核心步骤）
      |  → Grep 定位目标文件
      |  → Read 3-5 个核心文件，理解现有代码模式
      |  → Grep import 分析影响范围
      |  → 提取具体的代码风格锚点
      |
[第3步] Quest 拆分 + 排序
      |  → 列出变更清单（每个文件级操作）
      |  → 依赖拓扑排序
      |  → 原子化分组
      |  → 复杂度校准（防粒度不均）
      |
[第4步] 生成 Quest Map
      |  → 标准 7 字段格式
      |  → 每个 Quest 带变更清单 + 可执行验收
      |
[第5步] 自验证评分
      |  → 10 项评分，< 7 分必须修改
      |
[第6步] 防幻觉校验
      |  → Glob 验证所有文件路径
      |  → 技术一致性检查
      |  → 验收命令可执行性检查
      |
[第7步] 输出 + 等待确认
```

## 调用 quest-designer

```
Agent({
  subagent_type: "quest-designer",
  prompt: "你是 quest-designer。

【用户需求】[需求描述]
【技术栈】[语言 + 框架]
【完整能力清单】[从 /auto PHASE 1 传入，或此处现场扫描]
【现有代码文件】[src/ 下文件列表]

严格按 quest-designer 的 7 步工作流执行。
输出 Quest Map，等待用户确认。"
})
```

## Quest 标准输出格式

```markdown
# 《[项目/功能名称] 闯关大纲》

## 阶段 [X]：[阶段名称]
> [一句话说明本阶段解决什么问题]

### Quest [X.Y]：[具体动作描述]

🎯 **目标**：[具体到代码动作级别的一句话]
🛠️ **能力**：[Agent/Skill/MCP/直接编码]
💡 **选择理由**：[为什么选这个，为什么不选其他]
📁 **代码风格参考**：[具体文件 + 具体要模仿的模式]
🚫 **边界限制**：[具体禁止的文件/模块/操作]
📦 **变更清单**：[+] 新增 / [~] 修改 / [-] 删除 的具体文件
✅ **验收标准**：可粘贴执行的命令 + 预期输出
🔗 **依赖**：前置 Quest 编号
```

## 使用示例

### 简单需求
```
/auto:quest 给 SysUser 模块新增"重置密码"功能
```
输出：3-5 个 Quest，覆盖 DTO → Service → Controller → 测试

### 复杂需求
```
/auto:quest 实现完整的电商订单系统
技术栈：Spring Boot + MyBatis Plus + React
```
输出：20-30 个 Quest，分 5-6 个阶段

### 已有项目扩展
```
/auto:quest 在现有项目基础上增加评论系统
代码风格参考 src/modules/order 的目录结构
```
输出：每个 Quest 的 📁 都引用 src/modules/order 的具体模式

## 与 /auto 的关系

| 场景 | 推荐用法 |
|------|---------|
| 只看规划不执行 | `/auto:quest [需求]` → 输出 Quest Map |
| 规划 + 执行 | `/auto [需求]` → 内部调用 quest-designer → 逐关执行 |

**核心定位**：`/auto:quest` 只出蓝图不写代码，`/auto` 负责按蓝图施工。

---

## 执行协议

Quest Map 确认后的执行方式：

```
# 逐关执行（推荐）
/auto 执行 Quest 1.1

# 批量执行一个阶段
/auto 执行阶段 1

# 全自动（/auto 内部路径）
/auto 实现订单系统
→ PHASE 1 发现 → PHASE 2 Quest Map → PHASE 3 逐关执行
```

### 衔接规则

1. 执行时必须遵守 Quest 的 🚫 边界限制
2. 每个 Quest 完成后输出验收标准供检查
3. 验收失败只修复当前 Quest，不跳关
4. Quest 之间有依赖时自动检测是否满足
