# /auto:quest — 闯关大纲设计

> **输入一份需求，输出一套 PM 可验收的渐进式闯关教案**

`/auto:quest` 将复杂的商业需求（PRD）拆解为原子化的闯关大纲（Quest Map），让不写代码的 PM 也能逐步验收开发进度。

---

## 核心价值

- **PM 可验收**：每个关卡都有肉眼可见的黑盒验收标准
- **防止迷失**：严格防超纲，每关只做一件事
- **风格统一**：每关强制指定代码风格参考文件
- **第三方友好**：API 接入单独成关，降低认知负荷

---

## 执行流程

```
用户输入 PRD 需求
      ↓

├─────────────────────────────────────────────────────────┤

│  • 扫描目录结构，找到代码风格锚点                        │
│  • 读取 CLAUDE.md / .claude/rules/ 项目规范              │
└─────────────────────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────────────────────┐
│  第2步：需求反问（如有缺失信息）                          │
├─────────────────────────────────────────────────────────┤
│  • 技术选型约束？部署目标？数据库选型？                  │
│  • 移动端适配？第三方服务偏好？                          │
│  • 现有代码风格锚点？                                    │
│  • 如信息齐全，跳过此步                                  │
└─────────────────────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────────────────────┐
│  第3步：调用 quest-designer Agent 生成闯关大纲            │
├─────────────────────────────────────────────────────────┤
│  • 应用四大原则：原子化递进 + 黑盒验收 + 胶水编程 +      │
│    代码风格继承                                          │
│  • 输出标准 Quest Map 格式                               │
└─────────────────────────────────────────────────────────┘
      ↓
┌─────────────────────────────────────────────────────────┐
│  第4步：输出完整 Quest Map                                │
├─────────────────────────────────────────────────────────┤
│  • 分阶段 → 分关卡 → 每关带验收标准                     │
│  • PM 可直接按 Checklist 逐条验收                        │
└─────────────────────────────────────────────────────────┘
```

## 使用示例

### 示例 1：简单需求

```bash
/auto:quest 实现一个用户注册登录系统，用 React + Supabase
```

输出 Quest Map：约 8-12 个关卡，覆盖注册表单 UI → 表单验证 → Supabase 接入 → 登录逻辑 → Token 管理 → 路由守卫

### 示例 2：复杂 PRD

```bash
/auto:quest 实现完整的电商订单系统，包含商品浏览、购物车、下单、支付、发货追踪
技术栈：Spring Boot + MyBatis + React
数据库：MySQL
部署：Railway
```

输出 Quest Map：约 25-35 个关卡，分 6 个阶段

### 示例 3：已有项目扩展

```bash
/auto:quest 在现有项目基础上增加评论系统
代码风格参考 src/modules/order 的目录结构
```

输出 Quest Map：每个关卡的 `📁 代码风格参考` 都指向 `src/modules/order` 的既有模式

## 与 /auto 的关系

`/auto:quest` 是 `/auto` 超级命令的**规划前置步骤**。

| 场景 | 推荐用法 |
|------|---------|
| PM 想看开发计划 | `/auto:quest [PRD]` → 输出 Quest Map 给 PM 审核 |
| 开发者接到任务 | `/auto [需求]` → 内部调用 Quest Map 拆解（不区分复杂度，所有任务统一执行） |
| PM 审核通过后开发 | 按 Quest Map 逐关执行 `/auto [Quest N 描述]` |

**核心定位**：`/auto:quest` 只出教案不写代码，`/auto` 负责按教案执行。

---

## Quest Map 执行协议

Quest Map 生成并经 PM 审核后，按以下方式逐关执行：

```
# 方式 1：逐关手动执行（推荐，PM 可逐步验收）
/auto 执行 Quest 1.1：初始化项目脚手架，使用 create-next-app

# 方式 2：批量执行一个阶段
/auto 执行阶段 1 的所有 Quest（脚手架与环境搭建）

# 方式 3：全自动执行（/auto 内部统一走此路径）
/auto 实现电商订单系统
→ 自动能力发现 → AI 推理 → 生成 Quest Map → 逐步执行每个 Quest → 每步验证
```

### 执行时的衔接规则

1. **/auto 接收到 Quest 描述时**，必须遵守该 Quest 的 `🚫 边界限制`，绝不超纲
2. **每个 Quest 完成后**，输出该 Quest 的 `✅ 验收标准` 供 PM 检查
3. **PM 验收失败时**，/auto 只修复当前 Quest，不跳到下一关
4. **PM 验收通过后**，自动进入下一个 Quest
5. **如果 Quest 之间有依赖**（如"等后端 API 就绪"），/auto 自动检测依赖是否满足
