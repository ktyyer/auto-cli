# PRD Writer — 需求文档写作方法论

> 把模糊的产品点子，一步步变成可以直接拿去开发的需求文档。

## 快速使用

```
/auto 我想做一个 AI 写作助手，帮我写 PRD
/auto 把"用户可以在App里记账"这个想法变成需求文档
/auto 优化现有的 PRD，补充验收标准
```

## 何时使用

**必须加载**：

- 用户提到"写 PRD"、"写需求文档"、"产品需求"、"产品创意"
- 用户有想法但不知道如何形成规范文档
- 用户想让 AI 帮忙梳理产品需求
- 用户需要从零开始生成 PRD、优化已有文档、增加新需求

## 激活摘要 (Activation Digest)

**检查清单** (checklist):

- [ ] 第一阶段(概念版 10min): 明确目标用户+痛点+核心功能, 产出 1 页 PRD
- [ ] 第二阶段(落地版 30min): 补充验收标准、边界场景、非功能需求
- [ ] 用户视角: 目标用户是谁？现在怎么解决？痛点在哪？
- [ ] 功能范围: 包含什么？不包含什么？（明确边界）
- [ ] 验收标准: "做完算什么样？" 用可验证的方式描述

**硬约束** (constraints):

- 概念版不超 1 页（强制简洁）
- 功能范围必须明确 exclude（不做什么）
- 验收标准必须可验证（不是"用户体验好"这种描述）

**输出模板** (output):

- 概念版 PRD → 落地板 PRD（参考 `prd-writer.references/template.md`）

**反模式** (anti-patterns):

- 跳过概念版直接写详细文档 → 方向未对齐就投入细节
- 只有功能描述无验收标准 → 开发完不知算不算完成

---

## 核心流程

```
用户想法 → 概念版(10分钟) → 落地板(30分钟) → 可执行PRD
              ↓                  ↓
          方向对齐            细节补全
```

---

## 模式一：从零开始生成 PRD

### 第一阶段：概念版（10分钟）

**目标**：快速得到一个简短的概念版 PRD，与 AI 对齐方向。

**三个提问视角**：

#### 1. 用户视角

- 目标用户是谁？
- 他们现在怎么解决这个问题？
- 他们到底痛在哪？
- 你想做的产品真的帮到他了吗？

> 这一步能帮我们排除掉很多自嗨的伪需求。

#### 2. 商业视角

- 这是一次性需求还是长期需求？
- 商业模式是什么？买断制还是订阅制？
- 我们能不能持续做下去？

> 这一步是为了看清楚我们能不能持续地做下去。

#### 3. 技术视角

- 产品形态是什么？（App/Web/小程序/插件）
- 能不能快速做出来？
- 开发成本能不能接受？

**输出**：一个很短的概念版 PRD + 待确认问题列表

---

### 第二阶段：落地板（30分钟）

**目标**：补完所有细节，得到可以直接开发的 PRD。

**五个必填要素**：

#### 1. 交互细节

- 点击后报错了怎么办？
- 用户等待过程怎么展示？
- 加载状态、空状态、错误状态？

> 很多人会写"这里做一个按钮，点击后提交"。AI确实能帮你设计一个能点的按钮，但问题在于点了之后报错了怎么办？用户等待的过程要怎么展示？这些异常情况，如果你不提前写，AI是不会主动帮你想的。

#### 2. 状态机

- 一个字段或按钮的所有状态
- 状态之间的转换关系

#### 3. 字段规范

- 字段是否必填？
- 长度限制？
- 校验规则？

#### 4. 文案规范

- 对用户展示的文案需要统一
- 按钮文案的统一规范

#### 5. 各种图

- 核心用户流程图
- 关键页面的线框图

> 如果提前把这些东西都约定好，后面的 coding 部分会顺很多。

---

## 模式二：优化已有文档

当用户有现有 PRD 但质量不佳时：

1. **分析现有文档**：找出遗漏的细节和结构问题
2. **补充五个必填**：交互细节、状态机、字段规范、文案规范、流程图
3. **输出一致性格式**：确保文档结构完整

---

## 模式三：增加新需求

当用户需要给现有产品添加新功能时：

1. **定位现有功能**：新需求与现有功能的关系
2. **影响分析**：新需求对现有流程的影响
3. **整合输出**：将新需求无缝接入现有 PRD

## PRD 标准结构

```markdown
# [产品名称] PRD

## 1. 产品概述

- 产品背景
- 核心价值
- 目标用户

## 2. 用户场景

- 用户故事
- 使用流程

## 3. 功能需求

### 3.1 核心功能

### 3.2 功能流程图

## 4. 交互细节

### 4.1 正常流程

### 4.2 异常情况

### 4.3 状态定义

## 5. 字段规范

- 字段名、类型、必填、校验规则

## 6. 文案规范

- 按钮文案、提示文案、错误文案

## 7. 视觉稿

- 线框图链接
- 流程图链接

## 8. 技术约束

- 依赖系统
- 性能要求
```

## 使用示例

**用户输入**：

> 我想做一个自媒体账号定位工具

**AI 提问**（概念版）：

> 1. 你的目标用户是谁？是新手博主还是成熟创作者？
> 2. 用户现在怎么解决账号定位问题？
> 3. 你的数据打算是从哪儿取？
>    ...

**用户回答后输出**：概念版 PRD + 待确认问题

**继续深化**（落地板）：

> 基于你的回答，我来补全细节...

**最终输出**：完整的可直接开发的 PRD

## 三种模式快速选择

| 模式     | 用户说...               | 执行流程                 |
| -------- | ----------------------- | ------------------------ |
| 从零生成 | "我想做XXX"             | 概念版 → 落地板          |
| 优化文档 | "帮我优化这个PRD"       | 分析 → 补全五要素 → 输出 |
| 增加需求 | "给这个产品加个XXX功能" | 定位 → 影响分析 → 整合   |
