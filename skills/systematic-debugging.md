# Systematic Debugging — 系统化调试方法论

> 基于 [obra/superpowers/systematic-debugging](https://github.com/obra/superpowers/tree/main/skills/systematic-debugging) 适配。
> 核心原则：**先找到根因，再提出修复。** 违反此流程就是违反调试精神。

## 快速使用

```
/auto 页面加载后白屏，控制台报 TypeError
/auto 测试用例在 CI 通过但本地失败
/auto 接口响应从 200ms 变成 5s
```

---

## 铁律

```
没有完成根因调查，就不允许提出修复方案。
```

如果还没完成 Phase 1，你不能提出任何修复。

---

## 使用时机

**必须加载**：

- 遇到任何 bug、测试失败、意外行为
- 构建失败、CI 报错、性能问题
- 用户提到 debug/调试/排查/定位/修 bug/修复错误

**尤其在这些情况下必须遵守**：

- 时间紧迫时（紧急情况让人想猜答案）
- "看起来只要改一下 X" 的直觉
- 已经尝试了多个修复但都失败了
- 上一个修复没起作用
- 没有完全理解问题

**不要跳过**：

- 问题看起来很简单（简单 bug 也有根因）
- 你很着急（急于求成必定返工）
- 有人催你立刻修好（系统化方法比乱猜更快）

## 四阶段流程

**必须逐阶段完成，不可跳过。**

### Phase 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过任何错误或警告
   - 错误信息经常包含精确的解决方案
   - 完整阅读堆栈追踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？
   - 精确的复现步骤是什么？
   - 每次都会发生吗？
   - 无法复现 → 收集更多数据，不要猜

3. **检查最近变更**
   - 什么改动可能导致这个问题？
   - `git diff`、最近的 commits
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中的证据收集**

   **当系统有多个组件（CI → build → signing，API → service → database）时：**

   **在提出修复之前，先添加诊断探针：**

   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传递
     - 检查每层状态

   运行一次收集证据，确定哪里断裂
   然后分析证据，定位故障组件
   最后深入调查该组件
   ```

5. **追踪数据流**

   **当错误在调用栈深处时：**

   完整的向后追踪技术：
   → 读取 `systematic-debugging.references/root-cause-tracing.md`

   **快速版：**
   - 错误值从哪里产生的？
   - 谁用错误值调用的？
   - 持续向上追踪直到找到源头
   - 在源头修复，不要在症状处修复

### Phase 2：模式分析

**在修复之前先找到模式：**

1. **找到正常工作的示例**
   - 在同一代码库中找到类似的正常代码
   - 什么工作了，什么坏了？

2. **与参考实现对比**
   - 如果在实现某个模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 在应用之前完全理解该模式

3. **识别差异**
   - 正常代码和异常代码有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"这不重要"

4. **理解依赖**
   - 这个组件还需要什么？
   - 需要哪些设置、配置、环境？
   - 它做了什么假设？

### Phase 3：假说与验证

**科学方法：**

1. **提出单一假说**
   - 明确声明："我认为 X 是根因，因为 Y"
   - 写下来
   - 要具体，不要含糊

2. **最小化测试**
   - 做最小的改动来验证假说
   - 一次只改变一个变量
   - 不要同时修多个东西

3. **验证后再继续**
   - 修好了？→ Phase 4
   - 没修好？→ 提出新假说
   - 不要叠加更多修复

4. **不懂就说不懂**
   - 说"我不理解 X"
   - 不要装懂
   - 寻求帮助
   - 做更多研究

### Phase 4：实施修复

**修复根因，不是症状：**

1. **创建失败测试用例**
   - 最简单的复现
   - 尽量用自动化测试
   - 修复前必须有测试

2. **实施单一修复**
   - 针对已确认的根因
   - 一次只改一处
   - 不要"顺便"改进其他东西
   - 不要捆绑重构

3. **验证修复**
   - 测试通过了？
   - 没有破坏其他测试？
   - 问题真正解决了？

4. **修复不生效时**
   - 停下来
   - 数一数：已经尝试了几次修复？
   - < 3 次：回到 Phase 1，用新信息重新分析
   - **>= 3 次：停下来质疑架构（见下方步骤 5）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **3+ 次修复失败：质疑架构**

   **表明架构有问题的模式：**
   - 每次修复都暴露新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实施
   - 每次修复在其他地方制造新症状

   **停下来质疑根本假设：**
   - 这个模式从根本上是否合理？
   - 是否在"凭惯性坚持"？
   - 应该重构架构还是继续修补症状？

   **在尝试更多修复之前，先与你的用户讨论**

   这不是假说失败——这是架构错误。

   修复后追加多层防御：
   → 读取 `systematic-debugging.references/defense-in-depth.md`

## 红旗信号 — 必须停下来走流程

如果你发现自己在想：

- "先快速修一下，回头再调查"
- "试试改 X 看看行不行"
- "同时改多个地方，跑一下测试"
- "跳过测试，我手动验证"
- "大概是 X，让我修一下"
- "没完全理解但这样可能行"
- "模式说 X 但我要换个方式"
- "主要问题是这些：[列出修复方案，但没有调查]"
- 提出方案前没有追踪数据流
- **"再试一次修复"**（已经失败了 2+ 次）
- 每次修复都在不同位置暴露新问题

**以上任何一条都意味着：停下来，回到 Phase 1。**

**如果 3+ 次修复失败：** 质疑架构（见 Phase 4.5）

## 常见借口反驳

| 借口                       | 事实                                          |
| -------------------------- | --------------------------------------------- |
| "问题很简单，不需要流程"   | 简单问题也有根因。流程对简单 bug 来说也很快。 |
| "紧急情况，没时间走流程"   | 系统化调试比乱猜乱试更快。                    |
| "先试这个，然后调查"       | 第一次修复定了基调。从一开始就做对。          |
| "确认修复有效后再写测试"   | 没测试的修复不可靠。先写测试来证明。          |
| "同时修多个地方省时间"     | 无法隔离哪个修好了。还会引入新 bug。          |
| "参考太长了，我按理解来"   | 一知半解必定出 bug。完整阅读。                |
| "我看到问题了，修一下"     | 看到症状 != 理解根因。                        |
| "再试一次"（已失败 2+ 次） | 3+ 次失败 = 架构问题。质疑模式，不要继续修。  |

## 验收标准

- [ ] 每次修复任务执行了 Phase 1 的至少 3 个步骤（读错误、复现、查变更）
- [ ] 没有在未完成根因调查的情况下直接提出修复方案
- [ ] 修复方案是单一修改，不是多个改动的捆绑
- [ ] 修复前有失败测试用例（Phase 4 Step 1）
- [ ] 3+ 次修复失败时触发了架构质疑讨论
