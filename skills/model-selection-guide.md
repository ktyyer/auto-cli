# Model Selection Guide -- 模型选择指南

> 选对模型 = 省 3 倍成本 + 提 2 倍速度。本 Skill 基于 Addy Osmani 2026 工作流实践，
> 帮你在 Haiku、Sonnet、Opus 之间做出正确选择。

---

## 模型对比速查

| 维度 | Haiku | Sonnet | Opus |
|------|-------|--------|------|
| **成本** | 1x（最低） | 3x | 15x |
| **速度** | 最快 | 快 | 慢 |
| **推理深度** | 基础 | 强 | 最强 |
| **上下文理解** | 简单上下文 | 复杂上下文 | 最复杂上下文 |
| **编码能力** | Sonnet 的 90% | 最佳编码 | 与 Sonnet 编码相当 |
| **最佳用途** | 轻量、高频任务 | 主要开发 | 深度推理 |

## 具体场景映射

### Haiku 适用场景

| 场景 | 示例 | 原因 |
|------|------|------|
| Agent Worker | tdd-guide 编写测试 | 高频调用，成本敏感 |
| 结对编程 | 逐行审查代码 | 速度快，交互流畅 |
| 简单修改 | 修复拼写/格式 | 能力足够，无需强推理 |
| 批量操作 | 批量重命名/重构 | 并发执行，成本可控 |
| 文档生成 | JSDoc 注释补充 | 格式固定，质量稳定 |

### Sonnet 适用场景

| 场景 | 示例 | 原因 |
|------|------|------|
| 主开发流程 | 实现新功能 | 编码能力最佳 |
| Agent 编排 | quest-designer 分析 | 平衡推理深度与成本 |
| 调试 | 定位并修复 Bug | 理解复杂上下文 |
| 代码审查 | 审查 PR 变更 | 准确识别问题 |
| 重构 | 拆分大函数 | 理解依赖关系 |

### Opus 适用场景

| 场景 | 示例 | 原因 |
|------|------|------|
| 架构设计 | 微服务拆分方案 | 需要深度推理 |
| 安全审计 | 全量安全审查 | 不能遗漏任何风险 |
| 性能分析 | 定位性能瓶颈 | 需要系统级思考 |
| 技术选型 | 评估 3+ 候选方案 | 需要全面权衡 |
| 复杂 Bug | 跨系统数据不一致 | 需要追踪长链路 |

## 成本优化策略

### 策略 1：分层调用

```
Opus（规划） → Sonnet（实现） → Haiku（测试/文档）
```

复杂任务用 Opus 分析，输出方案后交给 Sonnet 编码，编码后交给 Haiku 写测试和文档。

### 策略 2：Haiku 先行

先让 Haiku 尝试，如果质量不满意再升级到 Sonnet。大多数简单任务 Haiku 足够。

### 策略 3：上下文匹配

上下文窗口最后 20% 避免启动复杂任务。此时用 Haiku 做收尾工作（提交、文档、小修复）。

## 常见错误

| 错误 | 正确 |
|------|------|
| 所有任务都用 Opus | 90% 任务用 Sonnet/Haiku |
| 怕质量差全用 Opus | Sonnet 编码能力已最佳 |
| 简单任务也用 Sonnet | Haiku 处理轻量任务性价比最高 |
| 不考虑上下文长度 | 长上下文任务优先选 Sonnet/Opus |
