# Brainstorming — 方案探索方法论

> 借鉴 [Superpowers/obra](https://github.com/obra/superpowers) 的 `brainstorming` skill。
> 核心原则：**实现策略不是 AI 偷偷选的**。AI 主动列 2-3 个方案，让用户选方向 — 然后再写一行代码。

## 快速使用

```
/auto 用 Spring Boot 实现用户登录
/auto 给项目加缓存
/auto 实现导出 PDF 功能
```

以上三个需求都会触发本 skill — 因为「OAuth/JWT/Session」「Redis/本地内存/数据库」「pdfkit/iText/wkhtmltopdf」都是合理但不同的实现路径。

---

## 使用时机

**必须加载**（策略 = 实现 / 重构）：

- 同一需求存在 ≥ 2 个合理技术方案（框架/库/协议）
- 涉及架构决策：同步 vs 异步、关系库 vs 文档库、新建 vs 重构、单体 vs 微服务
- 用户未指定技术选型，仅给目标
- 性能/可扩展性 vs 简洁性需要权衡

**不要触发**：

- 策略 = 探索 / 修复（无需多方案对比）
- 用户已显式指定方案：`用 OAuth 实现登录` / `用 Redis 加缓存`
- 单值修改 / bug 修复 / 文档微调
- 需求本身就是单路径（如"调整 nginx 超时"）

## 激活摘要 (Activation Digest)

**检查清单** (checklist):

- [ ] 触发判定：策略 ∈ {实现, 重构} 且 ≥ 2 条合理实现路径
- [ ] 列 2-3 方案，每个含：名称、核心做法、利、弊、适用场景、典型反例
- [ ] 以 200-300 字小节呈现，**禁止表格压缩**（小节让用户真读，表格扫一眼跳过）
- [ ] 给出 AI 推荐 + 推荐理由（不强制接受）
- [ ] 用 `AskUserQuestion` 让用户在 2-3 方案 + "其他（请补充）" 中选择
- [ ] 用户选定后写回 RouteDecision.selection.notes + QuestMap.decisionNotes
- [ ] 列举后 trade-off 仍不明朗（给不出有依据的推荐）→ 升级调用 `plan-ensemble` skill，不强行让用户选

**硬约束** (constraints):

- 必须列 ≥ 2 个方案，**禁止只列一个**（即使有"明显最优"也要列对比项）
- 每个方案必须含"反例" — 什么场景下它不是好选择
- 方案对比不超过 3 个 — 多了用户疲劳
- 用户选定后不得"自作主张换方案"

**输出模板** (output):

```markdown
## 方案探索

### 方案 A：<名称>

<200-300 字：核心做法 + 利 + 弊 + 适用场景 + 反例>

### 方案 B：<名称>

<同上>

### 方案 C：<名称>（可选）

<同上>

### AI 推荐

<推荐 A/B/C 中哪个，理由 1-2 句>

### 请选择

<AskUserQuestion: A / B / C / 其他（请补充）>
```

**反模式** (anti-patterns):

- "我觉得 X 方案明显最好，就用它" → 偷偷做选择，用户没参与
- 只列一个方案 + 列出利弊 → 不是 brainstorming，是 self-justify
- 列方案不给推荐 → 把决策完全甩给用户，AI 价值减半
- 表格化方案对比 → 用户扫一眼跳过，没真消化每个选项

## 核心流程（4 步）

### 第一步：触发判定

确认满足全部条件：

| 条件                   | 例                                              |
| ---------------------- | ----------------------------------------------- |
| 策略 ∈ {实现, 重构}    | ✓                                               |
| 用户需求明确（非歧义） | ✓ — 不是 requirement-clarifier 处理的"模糊需求" |
| ≥ 2 条合理实现路径     | ✓ — 登录有 OAuth/JWT/Session 三种               |
| 用户未指定方案         | ✓ — 没说"用 OAuth"                              |

任一项 ✗ → 跳过 brainstorming，直接进 PLAN 后续节。

### 第二步：方案列举（2-3 个）

每个方案从 **核心做法 → 利 → 弊 → 适用场景 → 反例** 五维展开 200-300 字。

**关键**：每个方案的"反例"是质量保证 — 强迫自己想"什么时候它不好"。

### 第三步：AI 推荐 + 1-2 句理由

主动给推荐，但不强制：

```
我推荐 B，因为 [本项目已有 X 基础设施 / 团队熟悉 Y / 性能要求需要 Z]。
但如果你更看重 [反例条件]，A 是更稳的选择。
```

### 第四步：用户确认 + 写回

用 `AskUserQuestion` 让用户选定。

选定后**强制写回**：

- `RouteDecision.selection.notes`：方案 X 被选定的核心理由
- `QuestMap.decisionNotes`：方案 X 的核心做法摘要 + 哪些细节交给后续 Quest

## 与 requirement-clarifier 的区别

两者**互补**，按下表分工：

| 维度     | requirement-clarifier             | brainstorming                   |
| -------- | --------------------------------- | ------------------------------- |
| 触发场景 | 需求**模糊**（"差不多" / "应该"） | 需求**明确但实现多路径**        |
| AI 角色  | 提问者（问用户问题）              | 分析师（输出 2-3 方案让用户选） |
| 用户回答 | 回答澄清问题                      | 在已分析的方案中选定            |
| 产出     | 澄清后的需求                      | 选定的方案 + 写回 decisionNotes |
| 上下游   | PLAN 早期（决定 RouteDecision）   | PLAN 中期（产出 QuestMap 前）   |

**决策树**：

```
用户提需求
   ↓
[策略 = 探索/修复] → 不走这两个 skill
   ↓ [策略 = 实现/重构]
[需求中有模糊词?] ─Yes→ requirement-clarifier
   ↓ No
[有 ≥ 2 条实现路径?] ─Yes→ brainstorming
   ↓ No
直接 PLAN
```

## 反模式（详细版）

### 反模式 1: AI 偷偷选最优方案

```
❌ "我用 OAuth 实现登录..."
✅ "登录有 3 种方案：OAuth / JWT / Session。我推荐 X 因为 Y。请选择？"
```

### 反模式 2: 列方案只为展示自己懂

```
❌ "方案 A: ...; 方案 B: ...; 方案 C: ...; 方案 D: ...; 方案 E: ..."
✅ 严格 2-3 个，每个 200-300 字。多了用户消化不了。
```

### 反模式 3: 把决策完全甩给用户

```
❌ "三个方案都各有利弊，你选吧。"
✅ "我推荐 B，因为 [本项目情况]。但如果 [反例条件]，A 更稳。"
```

## 与 plan-ensemble 的升级路由

列出 2-3 方案后若 trade-off 仍不明朗（利弊对冲，给不出有依据的推荐），不要强行让用户在信息不足时选择 — 升级调用 `plan-ensemble` skill：2-3 个异质视角隔离并行出计划草案，分歧点聚焦 + 评分矩阵合成唯一 QuestMap。

注意分工：brainstorming 是**方案级、用户选择**（OAuth vs JWT，串行列举）；plan-ensemble 是**视角级、并行竞争**（同一目标下不同规划哲学各自出草案，AI 合成）。策略=重构、或实现且复杂度 high 时，plan-ensemble 可不经 brainstorming 直接触发。

## 与 spec-driven 的协作

如果用户选定方案后还涉及具体接口/数据契约设计，转交 `spec-driven` skill 产出可执行 acceptance。
