# Refactoring Patterns — 安全重构方法论

> 借鉴 Martin Fowler《Refactoring》+ Anthropic 代码健康实践。核心原则：**重构不改变外部行为。** 改变行为是新功能，不是重构。

## 快速使用

```
/auto 拆分 utils.js（800 行），不改变行为
/auto 把重复的查询逻辑提取到公共方法
/auto 将 class component 重构为 function component
```

---

## 激活摘要 (Activation Digest)

**检查清单** (checklist):

- [ ] 重构前: 确保有测试保护网（regression 全绿）
- [ ] 小步提交: 每步只有一种重构手法，可独立回滚
- [ ] 不混入行为变更: 重构和新功能分离提交
- [ ] 验证: 重构后所有测试仍通过
- [ ] 大重构分批: 每次 ≤ 200 行变更

**硬约束** (constraints):

- 重构前 regression 测试必须全绿
- 禁止重构 + 新功能混在同一提交
- 大文件拆分只做结构重组，不改变逻辑
- 连续 3 次重构失败 → 停下来质疑架构

**输出模板** (output):

- 重构目标 → 影响范围 → 手法选择 → 测试状态 → 变更清单

**反模式** (anti-patterns):

- "顺便"优化逻辑 → 重构和新功能边界模糊，出问题难定位
- 无测试保护直接重构 → 改坏无人知
- 一次重写整个文件 → 风险过高

## 核心手法（按安全等级排序）

### 安全等级：高（IDE 可自动完成）

| 手法     | 说明                      | IDE 快捷键         |
| -------- | ------------------------- | ------------------ |
| 重命名   | 变量/函数/类 语义化重命名 | F2 (VS Code)       |
| 提取函数 | 一段逻辑抽成独立函数      | Ctrl+Shift+R       |
| 内联函数 | 函数体放回调用处          | Ctrl+Shift+I       |
| 移动函数 | 函数移到更合适的文件      | 拖拽 + import 更新 |
| 提取常量 | 魔数变命名常量            | Ctrl+Shift+R       |

### 安全等级：中（需手动 + 测试验证）

| 手法           | 说明                            | 注意             |
| -------------- | ------------------------------- | ---------------- |
| 拆分大文件     | 按职责拆分到多个文件            | 控制 import 循环 |
| 替换条件为多态 | if/switch → 策略模式            | 先建测试         |
| 引入参数对象   | 多参数 → 对象参数               | 不改变默认值     |
| 拆分循环       | 一个循环做多件事 → 多个独立循环 | 注意性能         |
| 移除死代码     | 删除未使用代码                  | git 已有历史     |

### 安全等级：需谨慎（较大的结构变化）

| 手法     | 说明                           | 前置要求            |
| -------- | ------------------------------ | ------------------- |
| 提取类   | 部分职责抽成新类               | 清晰的职责边界      |
| 替换算法 | 用更清晰算法替代               | 100% 测试覆盖原路径 |
| 分层重组 | Controller/Service/Mapper 调整 | 架构文档对齐        |
| 状态迁移 | Redux→Zustand 等               | 完整 E2E 保护       |

## 重构前安全检查清单

- [ ] 当前分支干净（`git status` 无未提交变更，或已暂存）
- [ ] 已有测试全绿（`npm test` 0 失败）
- [ ] 理解现有代码的职责（不是"看起来能改"）
- [ ] 建立回滚点（`git commit -m "pre-refactor checkpoint"`）
- [ ] 确认重构范围（改哪些文件/不改哪些）

## 大文件拆分策略

```
原文件 (800 行) → 分析职责边界
  ├── 类型/接口 (→ types.ts)
  ├── 工具函数 (→ utils.ts)
  ├── 核心逻辑 (→ 保留在主文件)
  ├── 子组件 (→ components/XXX.tsx)
  └── hooks (→ hooks/useXXX.ts)

拆分步骤:
1. 先拆类型/接口（最安全）
2. 再拆纯函数/utils（无副作用）
3. 再拆 hooks/子组件
4. 每次拆分后跑测试
5. 每次拆分单独提交
```

---

## 大规模重构的分批策略

| 批次 | 内容                      | 规模     |
| ---- | ------------------------- | -------- |
| P0   | 类型/接口整理 + 重命名    | ≤ 100 行 |
| P1   | 提取公共 utils/hooks      | ≤ 200 行 |
| P2   | 拆分大文件                | ≤ 300 行 |
| P3   | 架构调整（分层/模块重组） | ≤ 500 行 |

每批完成 → 测试 → 提交 → 再进下一批。拒绝"一次性全部改完"。
