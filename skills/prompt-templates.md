# Prompt Templates -- 场景化提示词模板库

> 与 `prompt-craft.md`（原则和方法论）互补，本文件提供可直接使用的结构化模板。
> 来源: Anthropic Prompt Engineering Guide, linux.do 社区精华, Claude Code 官方文档, Reddit r/ClaudeAI

---

## 模板 1：功能实现（结构化需求描述）

```
实现 [功能名称]，要求：

技术栈: [语言 + 框架]
文件范围: [涉及哪些文件/模块]
数据模型: [输入 → 处理 → 输出]
约束条件:
  - [约束 1]
  - [约束 2]
参考风格: [现有文件路径]（保持代码风格一致）
测试要求: [测试策略]
```

**为什么有效**: 将隐含假设全部显式化，AI 不需要猜测你的意图。

**来源**: Anthropic 官方 "Be specific and detailed" 原则

## 模板 2：Bug 调试（结构化故障描述）

```
Bug 描述: [一句话概括现象]
复现步骤:
  1. [步骤 1]
  2. [步骤 2]
  3. [步骤 3]
期望行为: [应该发生什么]
实际行为: [实际发生了什么]
环境: [OS / Node版本 / 框架版本]
已尝试: [已经试过什么方法]
相关文件: [文件路径列表]
错误信息: [完整的错误堆栈]
```

**为什么有效**: 消除了 AI 需要反复询问"环境是什么？""完整的错误信息是什么？"的往返。

**来源**: linux.do 社区 "Claude Code 调试最佳实践" 整理

## 模板 3：代码审查（多维度评估）

```
对以下代码进行全面审查，关注以下维度：

1. 安全性: SQL 注入、XSS、敏感信息泄露、权限控制
2. 性能: N+1 查询、内存泄漏、不必要的循环
3. 可维护性: 命名、函数长度、圈复杂度
4. 错误处理: 异常是否被正确捕获和处理
5. 代码风格: 是否与项目既有风格一致

代码文件: [文件路径]
项目规范: [CLAUDE.md 路径]
```

**为什么有效**: 多维度指令让 AI 不会只关注某一方面而忽略其他。

**来源**: Claude Code 官方 code-review 命令设计

## 模板 4：架构设计（约束驱动推理）

```
设计 [系统/模块名称] 的架构方案。

需求:
  - [需求 1]
  - [需求 2]

约束:
  - 技术栈: [框架]
  - 团队规模: [N] 人
  - 预期用户量: [量级]
  - 性能要求: [指标]
  - 已有系统: [需要集成的系统]

请给出:
  1. 整体架构图（文字描述）
  2. 核心模块划分和职责
  3. 数据流说明
  4. 关键技术决策和理由
  5. 潜在风险和缓解方案
```

**为什么有效**: 约束条件让 AI 在合理的范围内推理，避免提出不切实际的方案。

**来源**: Anthropic "Give Claude context" 原则 + architect Agent 设计经验

## 模板 5：重构（渐进式改造）

```
重构 [模块/文件名称]，目标: [重构目标]

约束:
  - 保持所有现有测试通过（不可修改测试断言）
  - 保持公共 API 接口不变
  - 每次只改一个方面（不要一次改多个）
  - 每步改动后运行测试确认

重构策略:
  1. 先添加类型注解（如果没有）
  2. 提取大函数为小函数
  3. 消除重复代码
  4. 简化条件逻辑
  5. 优化数据结构使用

参考文件: [文件路径]
```

**为什么有效**: "每次只改一个方面"是防止重构失控的关键约束。

**来源**: Martin Fowler 重构原则 + Claude Code refactor-clean 命令经验

## 模板 6：文档生成（用户视角）

```
为 [模块/功能] 生成文档：

目标读者: [初学者 / 高级开发者 / 运维人员]
文档类型: [API 文档 / 使用指南 / 架构说明]

包含内容:
  1. 一句话概述
  2. 快速开始（3 步以内）
  3. 完整使用示例（含输入输出）
  4. 常见问题 FAQ
  5. 相关链接

不要:
  - 不要写 "Overview" 标题然后留空
  - 不要用 "等等" 或 "..." 省略内容
  - 不要假设读者了解项目内部实现
```

**为什么有效**: "不要" 系列指令封堵了 AI 生成空洞文档的常见模式。

**来源**: linux.do 社区 "让 Claude 写出高质量文档" 讨论

## 模板 7：快速修复（最小变更）

```
修复问题: [一句话描述]

文件: [文件路径]
错误: [错误信息]

要求:
  - 最小变更（只改必要的部分）
  - 不重构、不优化、不改进
  - 保持代码风格与周围代码一致
  - 修复后运行相关测试确认
```

**为什么有效**: "不重构、不优化"约束防止 AI "顺手"改一堆无关代码。

**来源**: Claude Code build-fix Agent 的设计原则
