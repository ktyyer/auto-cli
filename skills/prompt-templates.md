# Prompt Templates -- 高效 Prompt 模板库

> 写好 Prompt = 写好代码的一半。本 Skill 收录 15 个经过社区验证的高质量 Prompt 模板，
> 覆盖功能开发、Bug 修复、代码审查、架构设计等高频场景，可直接复制使用。

---

## 模板使用原则

1. **结构化 > 随意写** -- 用模板比自由发挥效果好 3-5 倍
2. **具体 > 模糊** -- "修复登录按钮点击无响应" > "有个 bug"
3. **带上下文 > 零上下文** -- 告诉 AI 你用了什么框架、什么版本
4. **带验收标准 > 没有标准** -- "测试通过 + 覆盖率 >= 80%" > "写好点"

---

## 模板分类

### A 类：功能开发（5 个）

#### T1: 新功能开发（标准版）

```
请用 [框架名] 实现以下功能：

需求：[具体描述]
技术栈：[语言 + 框架 + 版本]
项目结构：[简要描述或引用 REPO_MAP.md]

要求：
1. 遵循项目现有的编码风格
2. 先写测试再写实现（TDD）
3. 函数不超过 50 行，文件不超过 400 行
4. 所有公共方法有 JSDoc/docstring 注释

验收标准：
- [ ] 编译通过：[编译命令]
- [ ] 测试通过：[测试命令]
- [ ] 覆盖率 >= 80%
- [ ] 无 console.log/print 调试语句
```

#### T2: 新功能开发（精简版，适合小改动）

```
在 [文件名] 中 [新增/修改] [具体内容]。

上下文：
- 这个文件的作用是 [简述]
- 相关文件：[列出依赖]

约束：
- 不要修改 [列出禁止修改的部分]
- 保持 [现有行为/接口] 不变
```

#### T3: 前端组件开发

```
创建一个 [组件名] 组件：

功能描述：[详细描述用户交互流程]
设计参考：[描述样式/Figma 链接/截图说明]

Props 接口：
- [prop名]: [类型] - [描述] （[必填/可选]，默认值: [值]）

状态管理：
- [state名]: [类型] - [描述]

边界情况：
- 空数据状态
- 加载中状态
- 错误状态
- 超长文本

技术要求：
- 响应式设计（支持手机/平板/桌面）
- 无障碍访问（aria 标签、键盘导航）
- 不引入新依赖
```

#### T4: API 接口开发

```
实现 [接口名] API：

路由：[HTTP 方法] [路径]
请求参数：[列出参数、类型、校验规则]
响应格式：[列出响应字段]

业务逻辑：
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

安全要求：
- 参数校验（不信任客户端）
- SQL 注入防护（参数化查询）
- 权限检查（@PreAuthorize 或中间件）

错误处理：
- 400: [参数错误场景]
- 401: [未认证场景]
- 404: [资源不存在场景]
- 500: [服务器错误场景]
```

#### T5: 数据库相关开发

```
为 [实体名] 实现数据访问层：

表结构参考：[描述或引用已有 Entity]
CRUD 操作：[列出需要的增删改查操作]

查询需求：
- 分页查询：支持按 [字段] 排序和 [字段] 过滤
- [其他特殊查询]

性能考虑：
- 避免 N+1 查询
- 大量数据使用分页而非全量查询
- 适当使用索引（在 [字段] 上）

MyBatis Plus 规范：
- 使用 LambdaQueryWrapper（不要硬编码字段名）
- 复杂 SQL 写在 XML 中
- 不使用字符串拼接 SQL
```

### B 类：问题修复（3 个）

#### T6: Bug 修复（结构化）

```
修复以下 Bug：

症状：[具体的错误信息或异常行为]
复现步骤：
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

期望行为：[正确的行为描述]
实际行为：[错误的当前行为]

相关文件：[列出可能相关的文件]
最近变更：[最近是否有相关代码被修改]

修复要求：
- 找到根因再修复（参考 root-cause-tracing Skill）
- 添加回归测试（确保不会复发）
- 不引入新 Bug
```

#### T7: 构建错误修复

```
修复以下构建错误：

错误信息：
```
[粘贴完整错误输出]
```

环境信息：
- Node.js / JDK / Python 版本: [版本]
- 包管理器: [npm/pnpm/yarn/pip/maven]
- 操作系统: [Windows/macOS/Linux]

已尝试的修复：
- [列出你已经尝试过的方法]

项目依赖变更：
- 最近是否升级了依赖？ [是/否，列出变更]
```

#### T8: 性能问题修复

```
解决性能问题：

现象：[描述具体表现，如 "列表加载超过 5 秒"]
影响范围：[影响多少用户/数据量]
测量方式：[如何重现和测量]

分析方向：
- [ ] 数据库查询是否有 N+1 问题？
- [ ] 是否有不必要的全量查询？
- [ ] 是否有内存泄漏？
- [ ] 是否有阻塞操作？
- [ ] 前端是否有不必要的重渲染？

约束：
- 不改变公共 API 接口
- 优化后提供性能对比数据（优化前 vs 优化后）
```

### C 类：质量保障（4 个）

#### T9: 代码审查

```
请审查以下代码变更：

变更范围：[文件列表或 git diff]
变更目的：[为什么要做这个改动]

审查重点：
- [ ] 逻辑正确性
- [ ] 边界条件处理
- [ ] 错误处理完整性
- [ ] 安全风险（注入、XSS、密钥泄露）
- [ ] 性能影响
- [ ] 代码可读性和可维护性
- [ ] 测试覆盖是否充分

输出格式：
| 严重程度 | 文件:行号 | 问题描述 | 修复建议 |
|---------|----------|---------|---------|
```

#### T10: 安全审查

```
对以下文件/模块进行安全审查：

目标：[文件列表]
关注点：
- 输入验证（是否所有用户输入都经过校验）
- 认证授权（是否有未保护的接口）
- 数据保护（敏感数据是否加密存储和传输）
- 注入防护（SQL/NoSQL/XSS/命令注入）
- 密钥管理（是否有硬编码密钥）
- 依赖安全（是否有已知漏洞的依赖）

输出格式：
| 风险等级 | 类型 | 位置 | 描述 | 修复方案 |
|---------|------|------|------|---------|
```

#### T11: 测试编写

```
为 [文件名] 编写单元测试：

被测模块功能：[简述]
测试框架：[vitest/jest/pytest/JUnit]

测试范围：
- 正常路径：[列出正常场景]
- 边界条件：[列出边界情况]
- 异常路径：[列出错误场景]
- 并发/异步场景（如适用）

Mock 策略：
- [需要 mock 的外部依赖]

覆盖率目标：>= 80%

命名规范：
- 测试文件: [命名模式]
- 描述格式: "should [预期行为] when [条件]"
```

#### T12: 重构

```
重构 [文件名/模块名]：

重构目标：[描述期望的改进]
当前问题：[描述当前代码的痛点]

重构范围：
- 允许修改: [列出可以改的部分]
- 不可修改: [列出不能改的公共接口]

约束：
- 保持所有现有测试通过
- 不改变公共 API 行为
- 每次只做一种重构（小步前进）
- 每步重构后运行测试验证

重构步骤（建议）：
1. [先做什么]
2. [然后做什么]
3. [最后做什么]
```

### D 类：设计规划（3 个）

#### T13: 架构设计

```
为 [系统/模块名] 进行架构设计：

业务背景：[描述业务需求和约束]
技术约束：[性能要求、可用性要求、数据量级]
现有架构：[简述或引用已有架构文档]

设计要求：
- 模块划分和职责定义
- 数据流和接口定义
- 错误处理策略
- 扩展性考虑

输出格式：
- 架构图（用 ASCII 或 Mermaid）
- 模块职责表
- 接口定义
- 关键决策和理由
```

#### T14: 技术选型

```
为 [功能需求] 进行技术选型：

需求描述：[具体需要实现什么]
候选方案：
1. [方案 A 名称] - [一句话描述]
2. [方案 B 名称] - [一句话描述]
3. [方案 C 名称] - [一句话描述]

评估维度：
- 功能完整性: [是否满足所有需求]
- 性能: [吞吐量/延迟/资源消耗]
- 社区活跃度: [GitHub Stars、最近更新]
- 学习成本: [团队是否熟悉]
- 维护成本: [长期维护难度]
- 生态兼容: [与现有技术栈的兼容性]

约束：
- 不引入与现有技术栈冲突的方案
- 优先选择社区活跃、文档完善的项目
```

#### T15: 需求分析

```
分析以下需求，输出实现计划：

原始需求：[用户描述]
业务背景：[为什么要做这个功能]

分析输出：
1. 需求拆解（拆为可独立实现的功能点）
2. 依赖分析（前置条件和外部依赖）
3. 风险点（可能的技术难点和业务风险）
4. 工作量估算（S/M/L/XL）
5. 实现顺序建议（依赖拓扑排序）
6. 验收标准（每个功能点的完成定义）
```

## Prompt 优化技巧

### 技巧 1：角色设定

在复杂任务前加上角色设定，效果显著提升：

```
你是一名 [角色描述，如 "10 年经验的 Java 架构师"]。
你的专长是 [具体领域]。
你的风格是 [描述工作方式]。

现在请完成以下任务：
[实际任务]
```

### 技巧 2：Few-Shot 示例

提供 1-2 个示例，让 AI 理解你的期望格式：

```
请按照以下格式输出代码审查结果：

示例：
| 严重程度 | 文件:行号 | 问题描述 | 修复建议 |
|---------|----------|---------|---------|
| 高 | auth.js:42 | 密码明文存储 | 使用 bcrypt 哈希 |

现在请审查：
[你的代码]
```

### 技巧 3：思维链

对于复杂分析任务，要求 AI 先思考再输出：

```
请按以下步骤分析：
1. 先列出所有可能的方案
2. 逐一评估每个方案的优缺点
3. 给出你的推荐方案和理由
4. 列出潜在风险和应对策略
```
