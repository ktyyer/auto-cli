# Performance Tips — 性能优化策略

> 融合 cost-optimizer 和 context-compression 两大能力
> 在不降低质量的前提下，将 AI 使用成本降低 50-80%

## Part 1: Token 成本优化

### OpusPlan 模型路由策略

不同阶段使用不同模型，在质量和成本之间取得最优平衡：

```text
规划阶段 → Claude Opus（慢但深度思考，只用一次）
执行阶段 → Claude Sonnet（速度快、成本低，反复迭代）
简单任务 → Claude Haiku（超低成本，适合格式化/摘要）
```

**实际成本对比**（同一复杂任务）：

| 策略 | 成本 | 质量 |
|------|------|------|
| 全程 Opus | $$$$ | 最高 |
| OpusPlan + Sonnet执行 | $$ | 接近最高 |
| 全程 Haiku | $ | 一般 |

### 五大节省策略

1. **模型路由**：按任务复杂度选择模型（见上方）
2. **上下文精简**：用锚定摘要替代完整文件（见 Part 2）
3. **批量处理**：合并相似请求，减少总调用次数
4. **缓存复用**：不变的数据只读一次，缓存结果
5. **精准检索**：只加载需要的代码片段，不全量读取

### 成本追踪

使用 `lib/cost-tracker.js` 追踪每次任务的 Token 消耗。

## Part 2: 上下文压缩

### 为什么需要压缩

Claude 的上下文窗口有限（200k token）。在复杂任务中：
- 对话历史不断积累 → token 耗尽
- 强制截断 → 丢失关键信息
- 重新开始 → 损失所有进度

### 锚定迭代摘要（Anchored Iterative Summarization）

**核心算法**：

1. **锚定**：识别关键信息（架构决策、文件修改、用户偏好）
2. **摘要**：用结构化摘要替代原始对话
3. **保留**：所有锚定点的内容不压缩
4. **迭代**：每完成一个阶段执行一次压缩

**压缩时机**：
- 上下文使用超过 60% 时主动触发
- 每完成一个阶段后执行
- 用户主动请求时

### 压缩模板

```markdown
## 任务摘要

### 已完成
- [x] 分析现有架构
- [x] 设计新方案（决策：使用微服务，理由：...）

### 关键决策
1. 选择 PostgreSQL 而非 MongoDB — 理由：需要事务支持
2. 使用 REST 而非 GraphQL — 理由：团队更熟悉

### 当前状态
- 正在实现：PaymentService
- 已修改文件：src/payment.js, src/config.js
- 待处理：编写测试，集成验证

### 未解决问题
- 支付回调的幂等性方案待定
```

## 综合建议

| 场景 | 成本策略 | 压缩策略 |
|------|---------|---------|
| 小改动（< 5 文件） | Haiku 全程 | 不需要压缩 |
| 中等功能（5-15 文件） | OpusPlan + Sonnet | 按阶段压缩 |
| 大型重构（15+ 文件） | OpusPlan + Sonnet + Haiku 辅助 | 主动压缩 + 检查点 |
| 超长任务（> 1h） | 同上 + 成本追踪 | 锚定摘要 + 状态持久化 |
