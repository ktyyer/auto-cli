# Reflection -- 反思能力

> 反思（Reflection）是 2026 年公认的 5 大 AI Agent 设计模式之一。让 AI 在完成一个动作后，
> 自动回顾自己的输出，发现不足并主动修正。社区实测数据显示，加入反思环节后输出质量提升 40%。

---

## 核心理念

1. **AI 不是一次性的** -- 它可以"做完 -> 回看 -> 改进"循环多轮
2. **自我评估比人类审查快** -- AI 在秒级内完成自检，不等人工
3. **反思需要明确标准** -- 没有"好"的定义，AI 就不知道"哪里不好"
4. **反思不等于重做** -- 反思是定向改进，不是推翻重来

---

## 三种反思模式

### 模式 1：即时反思（Inline Reflection）

在单次 Agent 输出后立即触发反思。

```
Agent 输出代码
  |
  v
反思检查点:
  - 是否满足需求中的每个验收条件？
  - 是否遵循项目的编码规范？
  - 边界情况是否已处理？
  - 错误处理是否完整？
  |
  v
IF 发现问题 -> 自动修正 -> 再次反思
IF 无问题 -> 进入下一步
```

**适用场景**：单文件修改、简单功能实现

**Auto CLI 集成**：PHASE 3 EXECUTE 中每个 Quest 完成后的自检步骤

### 模式 2：交叉反思（Cross-Review Reflection）

一个 Agent 的输出由另一个 Agent 评估。

```
Agent A 产出代码 -> Agent B 审查 -> 反馈给 Agent A -> 修正
```

**适用场景**：质量要求高的任务、安全敏感代码

**Auto CLI 集成**：
```
quest-designer 产出蓝图 -> architect 审查架构合理性 -> 反馈修正
tdd-guide 编写测试 -> code-reviewer 审查质量 -> 反馈修正
```

### 模式 3：累积反思（Cumulative Reflection）

在整个任务完成后，回顾全过程，总结经验。

```
所有 Quest 执行完毕
  |
  v
反思总结:
  - 哪些 Quest 的预判坑点命中了？
  - 哪些设计决策后来被推翻了？为什么？
  - 有没有更好的执行顺序？
  - 有没有可以固化为 Skill 的模式？
  |
  v
经验沉淀 -> Knowledge Steward
```

**适用场景**：大型任务（10+ Quest）、系统重构

**Auto CLI 集成**：PHASE 6 LEARN 阶段的反思总结

## 反思检查清单

### 代码级反思（每个 Quest 完成后）

```
[ ] 代码编译通过（零 error、零 warning）
[ ] 测试全部通过（覆盖率 >= 80%）
[ ] 命名清晰（不用 a/b/c/tmp/var）
[ ] 函数不超过 50 行
[ ] 无硬编码值
[ ] 错误处理完整（无空 catch 块）
[ ] 无安全漏洞（注入、XSS、密钥泄露）
[ ] 符合项目既有的编码风格
```

### 架构级反思（每 3-5 个 Quest 后）

```
[ ] 模块职责单一，没有"上帝类"
[ ] 依赖方向正确（高层不依赖低层实现细节）
[ ] 接口设计合理（参数不过多，返回值类型明确）
[ ] 性能无明显瓶颈（无 N+1 查询、无全表扫描）
[ ] 可扩展（新功能不需要大规模重构）
```

### 流程级反思（任务完成后）

```
[ ] Quest 拆分粒度是否合适？（太粗/太细）
[ ] 执行顺序是否最优？
[ ] 预判的坑点是否准确？
[ ] 合约定义是否完整？（有无遗漏的跨 Quest 依赖）
```

## 与 Auto CLI 的集成

### PHASE 2: REASON -- quest-designer 反思

quest-designer 在输出 Quest Map 后，执行一次即时反思：

```
1. 检查：每个 Quest 的合约是否完整定义？
2. 检查：依赖拓扑是否有循环？
3. 检查：验收标准是否每条可粘贴执行？
4. 检查：预判坑点是否基于代码分析（不是通用建议）？
5. IF 发现问题 -> 修正 Quest Map -> 再次反思
```

### PHASE 3: EXECUTE -- 逐关反思

每个 Quest 执行完毕后，执行即时反思：

```
1. 运行验收标准中的验证命令
2. IF 验证失败 -> 定位失败原因 -> 局部修正 -> 再次验证
3. IF 验证通过 -> 记录到 PHASE 6 知识沉淀
```

### PHASE 4: VERIFY -- 全量反思

所有 Quest 执行完毕后的全量门禁：

```
1. 编译检查
2. 测试检查（覆盖率 >= 80%）
3. 安全扫描
4. 代码风格检查
```

### PHASE 6: LEARN -- 累积反思

```
1. 统计：预判坑点命中率
2. 总结：哪些决策被推翻，为什么
3. 提炼：是否有可固化为 Skill 的模式
4. 沉淀：写入 Knowledge Steward
```

## 反模板警告

| 不要做 | 应该做 |
|-------|-------|
| 反思后推翻全部重来 | 只修正有问题的部分 |
| 反思标准模糊（"看看有没有问题"） | 使用结构化检查清单 |
| 每一行代码都反思（效率极低） | 按模块/函数粒度反思 |
| 反思后不修正就继续 | 发现问题必须修正后再继续 |
| 只用自我反思，不用交叉反思 | 关键模块用 code-reviewer 交叉审查 |
