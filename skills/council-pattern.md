# Council Pattern — 多 Agent 投票决策

> 重大决策不应该只有一个声音。Council 让 3 个独立视角各自分析，再通过结构化对比选出最优解。

## 适用场景

- 架构选型（微服务 vs 单体、数据库选型、缓存策略）
- 重构方案（有多种等价路径时）
- 技术选型（框架、库、部署方案）
- 任何"多个合理方案，需要权衡取舍"的决策

**不适用**：只有一种合理做法的任务（如修一个明确的 bug）。

---

## 工作流程

### 第 1 步：定义决策问题

```
明确以下信息：
  - 决策问题（一句话）
  - 约束条件（技术栈、性能要求、团队规模、预算等）
  - 评估维度（可维护性、性能、安全性、开发效率、学习成本等）
```

### 第 2 步：并行召集 3 个 Council 成员

**关键：3 个 Agent 必须并行启动，互不可见对方的输出。**

```
并行启动 3 个 Agent：

Agent({
  subagent_type: "architect",
  prompt: "你是 Council 成员 A（务实派）。
  决策问题：[问题描述]
  约束条件：[约束]

  你的视角：优先考虑开发效率和可维护性，偏好成熟稳定的方案。

  输出格式：
  ## 方案名称
  ### 核心思路（3 句话以内）
  ### 技术选择
  ### 优势（3 条）
  ### 劣势（3 条）
  ### 风险评估（高/中/低）
  ### 适用条件（什么情况下这个方案最优）"
})

Agent({
  subagent_type: "architect",
  prompt: "你是 Council 成员 B（性能派）。
  决策问题：[问题描述]
  约束条件：[约束]

  你的视角：优先考虑性能和可扩展性，愿意接受更高的复杂度换取更好的性能。

  输出格式：[同上]"
})

Agent({
  subagent_type: "architect",
  prompt: "你是 Council 成员 C（安全/风控派）。
  决策问题：[问题描述]
  约束条件：[约束]

  你的视角：优先考虑安全性、容错性和长期维护成本，偏好保守但可靠的方案。

  输出格式：[同上]"
})
```

### 第 3 步：结构化对比

收集 3 个方案后，生成对比矩阵：

```markdown
## Council 决策对比

| 维度 | 方案 A（务实派） | 方案 B（性能派） | 方案 C（安全派） |
|------|-----------------|-----------------|-----------------|
| 核心思路 | ... | ... | ... |
| 开发效率 | ★★★ | ★★☆ | ★★☆ |
| 性能 | ★★☆ | ★★★ | ★★☆ |
| 可维护性 | ★★★ | ★☆☆ | ★★★ |
| 安全性 | ★★☆ | ★★☆ | ★★★ |
| 学习成本 | 低 | 高 | 中 |
| 风险 | 低 | 中 | 低 |

## 共识点（3 个方案都同意的）
- ...

## 分歧点（需要决策的）
- ...

## 推荐
基于约束条件 [XXX]，推荐方案 [X]。
理由：[为什么这个方案在当前约束下最优]
```

### 第 4 步：输出决策 + ADR

```markdown
## 架构决策记录 (ADR)

**决策**：[选定方案]
**日期**：[日期]
**状态**：已决定

**上下文**：[决策问题 + 约束]

**考虑的方案**：
1. [方案 A] — [一句话]
2. [方案 B] — [一句话]
3. [方案 C] — [一句话]

**决策理由**：[为什么选这个]

**后果**：
- 正面：[优势]
- 负面：[需要接受的代价]
- 风险缓解：[如何应对劣势]
```

## 注意事项

- Council 只用于**重大决策**，不要对小问题启动（Token 成本 ~3x）
- 3 个成员必须**并行启动**，串行会导致后面的成员受前面的影响
- 对比矩阵必须包含**共识点和分歧点**，不只是罗列
- 最终推荐必须说明**在当前约束下为什么最优**，不是泛泛而谈
