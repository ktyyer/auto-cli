# PRD Writer — 需求文档写作方法论

> 把模糊的产品点子，一步步变成可以直接拿去开发的需求文档。

## 快速使用

```
/auto 我想做一个 AI 写作助手，帮我写 PRD
/auto 把"用户可以在App里记账"变成需求文档
/auto 优化现有的 PRD，补充验收标准
```

---

## 激活摘要 (Activation Digest)

**检查清单** (checklist):

- [ ] 第一阶段(概念版 10min): 明确目标用户+痛点+核心功能, 产出 1 页 PRD
- [ ] 第二阶段(落地版 30min): 补充验收标准、边界场景、非功能需求
- [ ] 用户视角: 目标用户是谁？现在怎么解决？痛点在哪？
- [ ] 功能范围: 包含什么？不包含什么？（明确边界）
- [ ] 验收标准: "做完算什么样？" 用可验证的方式描述

**硬约束** (constraints):

- 概念版不超 1 页（强制简洁）
- 功能范围必须明确 exclude（不做什么）
- 验收标准必须可验证（不是"用户体验好"这种描述）

**输出模板** (output):

- 概念版 PRD → 落地板 PRD（模板见 `prd-writer.references/prd-structure.md`）

**反模式** (anti-patterns):

- 跳过概念版直接写详细文档 → 方向未对齐就投入细节
- 只有功能描述无验收标准 → 开发完不知算不算完成

**必须加载**：用户提到"写 PRD"、"产品需求"、"产品创意"、有想法但不知如何形成规范

## 核心流程

```
用户想法 → 概念版(10分钟) → 落地板(30分钟) → 可执行PRD
              ↓                  ↓
          方向对齐            细节补全
```

---

## 模式一：从零生成 PRD

### 第一阶段：概念版（三个视角）

1. **用户视角** — 目标用户是谁？现在怎么解决？痛在哪里？真的帮到他了吗？
2. **商业视角** — 一次性还是长期？商业模式？
3. **技术视角** — 产品形态（App/Web/小程序）？开发成本？

→ 输出：1 页概念版 PRD + 待确认问题列表

### 第二阶段：落地板（五个必填要素）

1. **交互细节** — 点击报错怎么办？加载/空/错误状态？
2. **状态机** — 字段/按钮的所有状态和转换关系
3. **字段规范** — 必填/长度/校验规则
4. **文案规范** — 按钮和提示文案统一
5. **流程图** — 核心用户流程 + 关键页面线框图

---

## 模式二：优化已有文档

1. 分析现有文档 → 找出遗漏细节和结构问题
2. 补充五个必填要素
3. 输出一致性格式

## 模式三：增加新需求

1. 定位新功能与现有功能的关系
2. 影响分析（对现有流程的影响）
3. 整合输出

---

## 三种模式快速选择

| 模式     | 用户说...         | 执行流程                 |
| -------- | ----------------- | ------------------------ |
| 从零生成 | "我想做XXX"       | 概念版 → 落地板          |
| 优化文档 | "帮我优化这个PRD" | 分析 → 补全五要素 → 输出 |
| 增加需求 | "给这个产品加XXX" | 定位 → 影响分析 → 整合   |
