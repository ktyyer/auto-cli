# Diff-First (差异化优先) 编辑模式

在处理规模较大（数百行以上）的文件时，如果 AI 频繁地采用全文读取配合全文重新生成的策略，会带来三个严重后果：
1. **Token 消耗剧烈**：来回传输上百行未变动的代码，极大地浪费下行 Token 配额，且容易达到模型单次输出上限。
2. **上下文污染**：长篇代码容易让模型丢失细节，并在原有无需修改的地方意外诱发 bug 或改变代码格式缩进。
3. **变更极难审查**：用户和同事实在无法确认到底是仅仅改了一个变量，还是同时动了其它的三十个地方。

---

## 💡 Diff-First 原则

受到 **Aider** (search-and-replace block) 与 **Gemini CLI** 高级特性的启发，ai-max v5.0 的 Editor Agent 全面启用 Diff-First 编辑原则。

### 规则
除非是要创建全新的文件，或者是被要求的核心代码重构占比超过了整文件的 60%，否则 **必须总是** 运用 Diff 工具而不是全文覆盖。

### 核心运作与容错思路

```yaml
操作流程:
  1. 定位意图: 获取被要求修改的文件，和上下几行代码以确定上下文锚点。
  2. 生成 Diff 块:
       <<<<
       原始需要替换的精确代码
       ====
       新版本的业务代码
       >>>>
  3. 执行应用: 解析替换指令并精确地修改工作区文件。

故障容错:
  触发: 如果原始代码和本地文件不一致导致 Diff 失败。
  后备: Editor 会重试一次扩大寻找上下文锚点；若再失败则根据最新本地内容重新规划。
```

---

## Editor 的职责

在 `architect-editor` 的双模型协同内，Editor 会负责具体的细粒度修改。

如果需要同时在一份文件的新增导入头、修改类名、删除末尾方法：
- 不应该一次性通过字符串替换来覆写整个类。
- 应该组织成 3 个独立的 Diff 块序列连续执行。

```md
**Editor 执行报告:**
[Update] 替换 `UserController.java` (Imports) - 补全新增类的依赖
[Update] 替换 `UserController.java` (Line 40-42) - 修改 `deleteUser` 参数列表
[Drop] 删除 `UserController.java` (Line 126-140) - 移除无用代码
```

## 与 Git Auto Commit 结合
当这系列精确的修改落实到磁盘并且通过构建与测试关卡之后，Git 策略会立刻跟进，并以极端的粒度记录每一次差异化修改。

这一举措确立了能够实现真正的自动化工程开发。
