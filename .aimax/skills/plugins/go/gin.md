# Gin 开发助手（项目风格优先）

> 第一原则：**沿用项目现有 Go/Gin 风格**，不要硬套固定分层模板。

## 规则优先级

1. **项目现有代码**
2. **项目构建与静态检查配置**（`go.mod`、`golangci-lint`、`go fmt`）
3. **项目文档约定**
4. **本插件兜底建议**（仅无样例时）

## 执行流程

### Step 1: 先探测项目风格

生成前先确认：

- 路由组织方式（集中路由/分模块注册）
- 分层方式（handler/service/repo 或简化结构）
- 数据访问（GORM/sqlx/database/sql/DAO）
- 错误处理模式（error wrapping、自定义错误码、响应格式）
- DTO/模型组织和命名习惯
- 测试方式（table-driven、mock 方案）

### Step 2: 按样本对齐

至少参考同模块 2-3 个已有文件，对齐：

- 包结构与命名
- 请求绑定和参数校验方式
- 响应结构和状态码策略
- 日志和错误上报方式

### Step 3: 最小增量实现

- 只实现当前需求，不顺手全量重构
- 不把现有数据访问栈强制迁移到其他栈
- 不引入项目未使用依赖

## 强制约束（不可违反）

1. 项目未使用 GORM 时，禁止默认输出 GORM 模板代码。
2. 项目已有统一响应结构时，禁止并行新增另一套结构。
3. 项目已有错误码体系时，禁止绕开该体系直接返回随意错误。
4. 项目已有路由分组习惯时，禁止改变既有 URL 结构风格。

## 输出要求

实现说明必须包含：

- 参考了哪些现有 Gin 文件
- 对齐了哪些项目约定
- 哪些约定不明确并需要确认

## 兜底建议（仅无样例时）

仅在项目缺少可参考代码时，使用以下默认建议：

- 清晰的 handler 与业务逻辑分离
- 明确的参数校验和错误处理
- 一致的 HTTP 状态码与响应结构
- 基础单元测试或 handler 测试
