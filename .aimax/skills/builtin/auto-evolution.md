# Auto Evolution — 通用自主演进闭环

> 从“完成任务”升级为“持续变好”

## 触发方式

```bash
# 通过 /aimax:auto 自动触发
/aimax:auto 持续迭代这个模块并建立回归防护

# 直接指定目标
/aimax:auto 为支付链路建立评估门禁并自动优化
```

## 演进目标定义

先明确三类约束，再开始改动：

1. 业务目标（如吞吐、成功率、体验）
2. 工程目标（如覆盖率、构建稳定性、复杂度）
3. 风险边界（如不破坏公共接口、可随时回滚）

---

## 通用演进流程

### 1) 建立基线（Baseline）

- 收集当前状态：测试通过率、覆盖率、构建结果、关键性能指标
- 读取项目现有约束（lint/test/style/architecture）
- 产出本轮对照快照

### 2) 生成最小补丁

- 先用仓库结构定位关键文件和调用链
- 优先做“最小增量改动”，每轮只解决一类问题
- 每个补丁附带“预期收益 + 回滚点”

### 3) 执行验证矩阵

```yaml
required:
  - build_or_compile
  - unit_tests
  - lint_typecheck
optional:
  - integration_tests
  - e2e_tests
  - security_checks
  - llm_eval_suite
```

### 4) 门禁判定

```yaml
gates:
  tests: "必须全部通过"
  regressions: "高危回归为 0"
  coverage: "不下降"
  perf: "达到目标或给出差距说明"
```

### 5) 失败恢复（最多 3 轮）

1. 缩小范围定位根因
2. 按“同方案微调 → 替代方案 → 回滚”重试
3. 输出人工决策建议，不盲目无限循环

### 6) 经验沉淀

- 记录成功策略：适用条件、收益、可复用模板
- 记录失败反例：触发条件、根因、规避手法
- 让下一轮默认继承这些经验

---

## 项目风格优先原则

```text
1) 优先读取项目已有代码与配置
2) 对齐当前项目的命名、分层、异常、测试约定
3) 仅在缺少样例时使用通用兜底建议
```

```markdown
## 🔁 演进结果

  - P95: -22%
```
