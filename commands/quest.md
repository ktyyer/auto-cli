# /auto:quest — 闯关大纲设计 v3

> **输入一份需求，输出一套精确到方法签名和代码片段的渐进式闯关教案**

`/auto:quest` 将复杂需求拆解为零歧义的 Quest Map。核心差异化：**合约驱动 + 代码片段锚定 + 风险分层**。

---

## v3 核心升级

- **合约系统**：跨 Quest 的接口协议，确保类型一致性（DTO 字段、方法签名、返回类型）
- **实现蓝图**：每个 Quest 包含方法签名/类骨架 + 反模式警告
- **风险分层**：🔴高/🟡中/🟢低风险分类，高风险配备额外护栏
- **代码片段锚定**：3-5 行实际代码作为复制模板
- **回滚方案**：每个 Quest 带 git 回滚命令
- **15 项自验证**：升级自 v2 的 10 项，通过线 10/15

---

## 执行流程

```
用户输入需求
      |
[第1步] 需求解析 + 变更范围预判
      |  → 关键词提取 + 预判涉及文件
      |
[第2步] 深度代码分析（核心步骤）
      |  → 分层读取 3-8 个核心文件
      |  → 全链路依赖分析（import + 配置 + 运行时 + 数据库）
      |  → 提取代码片段锚点
      |
[第3步] 合约驱动式 Quest 拆分
      |  → 变更清单（精确到方法级）
      |  → 先定义合约(CONTRACT)再拆分 Quest
      |  → 依赖拓扑排序
      |  → 风险分层（🔴高/🟡中/🟢低）
      |  → 复杂度校准
      |
[第4步] 生成 Quest Map（10 字段零歧义格式）
      |  → 🧩 实现蓝图 + 反模式警告
      |  → ⚠️ 风险等级
      |  → 🔗 合约引用 + 🔙 回滚方案
      |
[第5步] 合约一致性校验 + 15 项自验证
      |  → 跨 Quest 类型一致性校验
      |  → 路径即时校验
      |  → 每项 0-1 分，总分 15，通过线 10
      |
[第6步] 输出 + 等待确认
```

## Quest v3 标准输出格式

```markdown
# 《[项目/功能名称] 闯关大纲》

## 全局信息
**技术栈**：[语言 + 框架]
**建议执行模式**：[单Agent / Subagent并行 / Agent Teams]
**关键合约清单**：CONTRACT-1 (...), CONTRACT-2 (...)

---

## 阶段 [X]：[阶段名称]

### Quest [X.Y]：[具体动作描述]

🎯 **目标**：[具体到字段名+类型+注解的一句话]
🛠️ **能力**：[Agent/Skill/MCP/直接编码]
💡 **选择理由**：[为什么选 + 为什么不选其他]
⚠️ **风险等级**：[🔴高/🟡中/🟢低]
📁 **代码风格参考**：[文件路径] + 3-5 行可复制代码片段
🚫 **边界限制**：[禁止的文件/模块/模式/技术]
📦 **变更清单**：[+] 新增 / [~] 修改 / [-] 删除 的具体文件
🧩 **实现蓝图**：方法签名或类骨架 + 反模式警告
✅ **验收标准**：语义级验证命令 + 预期结果
🔗 **合约**：产出/消费的合约编号
🔙 **回滚方案**：git 回滚命令
🔗 **依赖**：前置 Quest 编号
```

## 使用示例

### 简单需求
```
/auto:quest 给 SysUser 模块新增"重置密码"功能
```
输出：2-4 个 Quest，合约驱动确保 Service/Controller 类型一致

### 复杂需求
```
/auto:quest 实现完整的电商订单系统
技术栈：Spring Boot + MyBatis Plus + React
```
输出：15-25 个 Quest，分 4-5 个阶段，含完整合约体系和风险评估

## 与 /auto 的关系

| 场景 | 推荐用法 |
|------|---------|
| 只看规划不执行 | `/auto:quest [需求]` → 输出 Quest Map |
| 规划 + 执行 | `/auto [需求]` → 内部调用 quest-designer v3 → 逐关执行 |

**核心定位**：`/auto:quest` 只出蓝图不写代码，`/auto` 负责按蓝图施工。

---

## 执行协议

Quest Map 确认后的执行方式：

```
# 逐关执行（推荐）
/auto 执行 Quest 1.1

# 批量执行一个阶段
/auto 执行阶段 1

# 全自动（/auto 内部路径）
/auto 实现订单系统
→ PHASE 1 发现 → PHASE 2 Quest Map → PHASE 3 逐关执行
```

### 衔接规则

1. 执行时必须遵守 Quest 的 🚫 边界限制
2. 执行时必须参考 🧩 实现蓝图中的方法签名和反模式警告
3. 每个 Quest 完成后按 ✅ 验收标准验证
4. 验收失败使用 🔙 回滚方案恢复，然后修复重试
5. Quest 之间有依赖时检查 🔗 合约是否满足
